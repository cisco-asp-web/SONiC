## Docker 101: The Basics and Architecture

### 1. Virtualization vs. Containerization

To a network engineer, a Virtual Machine (VM) is a familiar concept. A VM is a fully emulated computer. You have physical hardware, a Hypervisor (like ESXi, KVM, or Hyper-V), and then the Virtual Machine. The VM contains virtualized hardware (vCPU, vRAM, vNICs) and runs a complete, independent Guest Operating System (e.g., Windows, Ubuntu).
While VMs provide strong isolation, they carry significant computational overhead. Booting a VM means booting an entire OS kernel.

**Containers are fundamentally different. They are not VMs.**
A container is simply a standard Linux process (or a group of processes) that has been logically isolated from the rest of the system. 

*   **No Hypervisor**: Containers run directly on the physical host's operating system.
*   **Shared Kernel**: All containers on a host share the same underlying Linux kernel. If the host is running Debian kernel 5.10, every container is using that exact kernel to execute system calls.
*   **Speed & Efficiency**: Because there is no Guest OS to boot, starting a container takes milliseconds. You are merely launching a process. You can comfortably run hundreds of containers on a single switch or server.

### 2. The Magic of Linux: Namespaces and Cgroups

If containers are just processes, how are they isolated? Docker relies on two fundamental Linux kernel features: **Namespaces** and **Control Groups (cgroups)**.

#### 2.1 Namespaces (Isolation)
Namespaces restrict what a container can *see*. They create the illusion that the container is the only entity running on the system.

*   **PID (Process ID) Namespace**: Inside the container, your main application is assigned Process ID 1. However, to the host OS, that exact same process might be PID 15483. The container cannot see or kill processes outside its PID namespace.
*   **NET (Network) Namespace**: This is the most crucial for network engineers. A NET namespace gives the container an entirely isolated network stack. This includes its own interfaces (eth0), its own routing table (`ip route`), its own ARP table, and its own `iptables` / `nftables` rules.
*   **MNT (Mount) Namespace**: Isolates the filesystem mount points. The container has its own root directory (`/`) and cannot see the host's filesystem unless explicitly allowed.
*   **UTS (UNIX Time-Sharing) Namespace**: Allows the container to have its own hostname and domain name independent of the host.
*   **IPC (Inter-Process Communication) Namespace**: Isolates system V IPC and POSIX message queues.
*   **USER Namespace**: Allows a process to have root privileges inside the container but map to an unprivileged, standard user on the host system.

#### 2.2 Control Groups / cgroups (Resource Limiting)
While namespaces limit what a container can *see*, cgroups limit what a container can *use*. Without cgroups, a single runaway container could consume 100% of the host's CPU and crash the router.

*   **Memory cgroup**: Restricts the maximum amount of RAM the container can allocate. If it exceeds this, the kernel's OOM (Out Of Memory) killer terminates the container.
*   **CPU cgroup**: Limits CPU usage. You can allocate "CPU shares" (relative weight compared to other containers) or "CPU quotas" (hard limits, e.g., max 0.5 CPU cores).
*   **Blkio cgroup**: Throttles block I/O (disk read/write speeds), preventing a container from starving the disk.

### 3. Docker Engine Architecture

When we say "Docker", we are referring to a stack of technologies. Docker uses a client-server architecture:

1.  **Docker CLI (`docker`)**: The command-line tool you use (`docker run`, `docker ps`). It does not run containers itself; it merely sends REST API requests to the daemon.
2.  **Docker Daemon (`dockerd`)**: The background service running on the host OS. It listens for API requests and manages images, networks, and volumes.
3.  **containerd**: An industry-standard core container runtime. `dockerd` delegates the actual lifecycle management of containers to `containerd`.
4.  **runc**: The lowest-level tool. It is the CLI wrapper around the Linux kernel features (namespaces, cgroups). `containerd` calls `runc` to actually spawn the isolated process.

### 4. Images, Layers, and OverlayFS

*   **Image**: A read-only template containing your application, binaries, and libraries. Images are built using a `Dockerfile`.
*   **Layered Filesystem (UnionFS / OverlayFS)**: Images are not single massive files. They are composed of distinct, stacked layers. 
    *   Layer 1: Base OS (e.g., Debian 11 - 50MB)
    *   Layer 2: Install Python (30MB)
    *   Layer 3: Copy routing script (1MB)
    Docker stacks these layers so they appear as one unified filesystem. If you download another image that also uses Debian 11, Docker reuses Layer 1, saving massive amounts of disk space.
*   **The Container Read-Write Layer**: When you run a container from an image, Docker adds a thin, ephemeral **read-write layer** on top of the read-only image layers.
    *   **Copy-on-Write (CoW)**: If the container tries to modify a file that exists in a read-only lower layer, Docker copies the file up into the read-write layer before modifying it.
    *   **Ephemeral Data**: Because the read-write layer is tied to the lifecycle of the container, **any data written directly inside a container is permanently lost when the container is deleted**. (We solve this with Volumes, covered in 202).

### 5. Essential Docker CLI Mastery

*   **`docker ps`**: Lists running containers.
    *   `docker ps -a`: Shows all containers, including stopped ones.
    *   *Output highlights*: Container ID, Image Name, Command executed, Status (Up 2 hours, Exited), and Port mappings.
*   **`docker images`**: Lists locally cached images with their size and tag (version).
*   **`docker run -d --name my_router <image>`**: Creates and starts a container in the background (`-d` for detached).
*   **`docker exec -it <container_name> /bin/bash`**: The most important troubleshooting command. It executes a new process (`/bin/bash`) *inside* the existing container's namespaces. The `-it` flags attach your keyboard and terminal to it.
*   **`docker logs -f <container_name>`**: Tails the standard output (STDOUT/STDERR) of the container. If a routing daemon crashes, the reason is usually printed here.
*   **`docker stop <container_name>`**: Sends a SIGTERM to gracefully stop the container's main process.
*   **`docker rm -f <container_name>`**: Forcefully deletes a container and its ephemeral read-write layer.
*   **`docker top <container_name>`**: Shows the processes running inside the container (like Linux `top`).
*   **`docker stats`**: Provides a live, auto-updating stream of resource usage (CPU %, Memory, Network I/O) for all running containers.

### 6. The SONiC Connection: Why is SONiC Containerized?

SONiC (Software for Open Networking in the Cloud) completely abandons the legacy monolithic network OS architecture (where everything is baked into a single massive firmware image). 

SONiC runs a base Debian Linux OS on the switch hardware, and then runs almost all of its networking functions as individual Docker containers:
*   **`bgp` container**: Runs FRRouting (FRR) for BGP, OSPF, etc.
*   **`database` container**: Runs Redis, the central nervous system of SONiC where all state and configuration are stored.
*   **`swss` (Switch State Service) container**: The translation layer between the Redis database and the physical ASIC (broadcom, cisco, etc.).
*   **`syncd` container**: The only container that directly talks to the physical ASIC via the vendor's SDK (SAI - Switch Abstraction Interface).
*   **`snmp`, `lldp`, `telemetry`**: All run in their own containers.

**The Benefit**: Microservices architecture. If the LLDP daemon crashes or has a memory leak, it only affects the `lldp` container. You can run `docker restart lldp` from the SONiC shell. The container restarts in milliseconds, while BGP stays up and data plane traffic continues forwarding without dropping a single packet.

## Docker 202: Advanced Concepts & Deep Dive Networking

### 1. Deep Dive: Docker Networking for Network Engineers

Docker networking relies heavily on Linux kernel networking primitives. Understanding the packet walk is essential for troubleshooting.

#### 1.1 The Default Bridge Network (`docker0`)
When you install Docker, it creates a virtual Linux bridge interface named `docker0` in the host's default network namespace.

**The Topology:**
1.  **netns creation**: When you start a container without specifying a network, Docker creates a new Network Namespace for it.
2.  **veth pair**: Docker creates a Virtual Ethernet (veth) pair. Think of this as a virtual Ethernet patch cable.
3.  **Attachment**: Docker drops one end of the veth cable into the container (renaming it `eth0` and assigning an IP like `172.17.0.2/16`). It plugs the other end into the `docker0` bridge on the host OS.

**The Packet Walk (Outbound to Internet):**
1.  Container generates a packet destined for `8.8.8.8`.
2.  Packet hits the container's default gateway (the IP of the `docker0` bridge, e.g., `172.17.0.1`).
3.  Packet travels over the veth pair into the host's network namespace, arriving at `docker0`.
4.  The host routing table forwards the packet to the physical interface (`eth0`).
5.  **NAT/Masquerade**: Before leaving the physical interface, the packet hits the host's `iptables` POSTROUTING chain. Docker has automatically inserted rules to Masquerade (Source NAT) the packet. The source IP changes from `172.17.0.2` to the host's physical IP.

**The Packet Walk (Inbound / Port Publishing):**
When you run `docker run -p 8080:80 nginx`, you are telling Docker to listen on host port 8080 and forward to container port 80.
1.  Packet from internet hits the host's physical IP on port 8080.
2.  The packet hits the `iptables` PREROUTING chain (specifically the `DOCKER` chain).
3.  A Destination NAT (DNAT) rule rewrites the destination IP to `172.17.0.2:80`.
4.  The host routes the packet to the `docker0` bridge, down the veth pair, and into the container.

#### 1.2 Host Networking (`--net=host`)
This is crucial for network operating systems. When a container is run with `--net=host`, **Docker does not create a new network namespace.** 
The container shares the exact same network stack as the host OS.

*   **Result**: If you type `ifconfig` inside the container, you see the host's actual physical interfaces. 
*   **Why SONiC needs this**: The `bgp` container in SONiC must use host networking. It needs to establish BGP peering directly out of the physical switch ports (e.g., `Ethernet0`), and it needs to inject learned routes directly into the host's routing table (which the `swss` container then reads and pushes to the ASIC).
*   **Performance**: Host networking has zero NAT and zero bridging overhead, providing line-rate software performance.

#### 1.3 Macvlan and IPvlan Drivers
Used when you need containers to act like physical devices on your existing subnets, bypassing NAT entirely.

*   **Macvlan**: Takes a parent interface (e.g., `eth0` or a VLAN sub-interface like `eth0.100`) and creates virtual sub-interfaces, each with a randomly generated, unique MAC address. The container gets one of these interfaces. To the external physical switch, it looks like multiple physical devices are plugged into the same port.
*   **IPvlan**: Similar to Macvlan, but all containers share the parent interface's MAC address. The separation is done purely at Layer 3 (IP). This is useful if the upstream physical switch enforces port security (limiting MAC addresses per port).

### 2. Containerlab: How Virtual Network Labs Work

In this lab, we use **Containerlab**. It's vital to understand that Containerlab largely **ignores standard Docker networking (`docker0`)**.

Instead, Containerlab acts as an orchestration engine that manually builds router topologies using Linux kernel tools:
1.  **Read Topology**: Containerlab parses your YAML file defining routers and links (e.g., `Node A eth1` connects to `Node B eth2`).
2.  **Deploy Containers**: It tells Docker to start the router containers but instructs Docker *not* to attach them to any network (`--net=none`).
3.  **Manual Wiring**: Containerlab then executes Linux commands (`ip link add type veth`) to create custom veth pairs.
4.  **Namespace Injection**: Using `ip netns`, it pushes one end of the veth pair directly into Node A's namespace (renaming it `eth1`) and the other end into Node B's namespace (`eth2`).
5.  **Result**: You now have a direct, isolated, point-to-point Layer 2 link between the two containers. It perfectly simulates a physical patch cable. LLDP, BGP, and OSPF run over this link seamlessly.

### 3. Capabilities and Privileged Mode

By default, Docker implements strict security. It drops almost all Linux "capabilities" (granular root privileges). A standard container cannot modify routing tables, load kernel modules, or change the firewall.

*   **`CAP_NET_ADMIN`**: If a container runs a routing daemon (like FRR/Quagga), it requires permission to manipulate the kernel's routing table and interfaces. You must start the container with `--cap-add=NET_ADMIN`.
*   **`CAP_NET_RAW`**: Required if the container needs to craft raw sockets (e.g., for running `ping` or BFD).
*   **Privileged Mode (`--privileged`)**: Disables all namespace and cgroup security isolation. The container gets full, unhindered root access to the host kernel, all hardware devices in `/dev`, and the ability to load kernel modules. Network emulators (like the Cisco 8000 emulator or virtual SONiC switches) almost always require this to emulate hardware ASICs and manipulate the host's dataplane.

### 4. Storage: Volumes vs. Bind Mounts

Because container filesystems (the read-write layer) are ephemeral, data is lost on reboot. We use two methods to persist data:

1.  **Bind Mounts (`-v /host/path:/container/path`)**:
    *   Directly mounts a specific file or directory from the host OS into the container.
    *   **Lab Use Case**: Injecting configuration files. Containerlab will bind-mount your local `startup_config.cfg` into the container's `/etc/frr/frr.conf`.
    *   **SONiC Use Case**: The central config file `/etc/sonic/config_db.json` lives on the host and is bind-mounted into almost every SONiC container so they all share the exact same configuration truth.
2.  **Volumes**:
    *   Docker-managed storage located in `/var/lib/docker/volumes/`.
    *   Volumes are abstracted away from the host filesystem. They are preferred for database storage (like SONiC's Redis state) because Docker handles the file permissions and performance optimizations natively.

### 5. Advanced Troubleshooting Toolkit for Network Engineers

When things break in a containerized network, standard tools often fail. Here is your advanced toolkit:

#### 5.1 Docker Inspect
`docker inspect <container_name>` returns a massive JSON object detailing the container's configuration.
*   Find the exact IP address: Look for `NetworkSettings.IPAddress`.
*   Find the host PID: Look for `State.Pid`. (Crucial for `nsenter`).
*   Verify bind mounts: Look under `Mounts` to ensure your config files are actually mapped correctly.

#### 5.2 Packet Captures Inside Containers
If the container has `tcpdump` installed, you can execute it directly inside the isolated network namespace:
`docker exec -it <container_name> tcpdump -i eth1 -n -e -w /tmp/capture.pcap`
You can then use `docker cp` to pull the pcap out of the container to your host for Wireshark analysis.

#### 5.3 The Ultimate Tool: `nsenter` (Namespace Enter)
What if the container doesn't have `ping`, `ip route`, or `tcpdump` installed? (Many production images are stripped bare for security).

You can use the host OS's networking tools to troubleshoot the container's network namespace!
1.  Find the container's Process ID on the host:
    `docker inspect --format '{{.State.Pid}}' <container_name>`
    *(Let's assume the PID is 5555).*
2.  Use `nsenter` to execute the host's `tcpdump` binary, but place it inside the network namespace (`-n`) of PID 5555:
    `sudo nsenter -t 5555 -n tcpdump -i eth0`
3.  You can also check the routing table of the container using the host's `ip` command:
    `sudo nsenter -t 5555 -n ip route`

This command is practically magic for network troubleshooting in heavily containerized environments like SONiC or Kubernetes.

#### 5.4 Accessing SONiC Specific Daemons
In SONiC, you don't usually configure BGP by editing text files. You enter the routing shell.
Because FRR runs inside the `bgp` container, you execute the VTY shell command directly into the container:
`docker exec -it bgp vtysh`
This drops you into a Cisco-like CLI to check BGP neighbors, route maps, and OSPF databases, entirely isolating the command to the BGP daemon.
